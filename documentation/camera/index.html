-- initialize the camera, passing a settings table with optional parameters or setting them directly
camera = fg.Camera({lerp = 3, follow_style = 'lockon', target = followed_entity})
camera.debug_draw = true

-- update the camera
camera:update(dt)

-- draw the camera, attach/detach applies/removes camera transforms
camera:attach()
entities:draw()
camera:detach()


Description

The fg.Camera module handles basic things that a camera should handle, like translations, rotations and scaling, but also adds a bunch of high level features such as target tracking, lerp (delay between camera position and target position), lead (how much in front of the target the camera should be), deadzones and camera shakes. An instance fg.world.camera is automatically created on initialization, but you can also create your own instances doing camera = fg.Camera().

This module is based on hump.camera and has been modified for additional functionality and consistency. Also, check HaxeFlixel’s camera demo for an example on how lerp, lead and targetting modes work. This module copies HaxeFlixel’s camera pretty closely for those features.

Methods

:new(settings[table[optional]])
  • settings: a table containing all camera initialization settings, all values are optional

Possible settings:

  • x, y: the initial position of the camera, defaults to the center of the screen
  • bounds: a table in the format left = [number], top = [number], right = [number], down = [number]
  • follow_style: the follow style: 'lockon', 'screen', 'platformer', 'topdown' or 'topdown-tight', defaults to 'lockon'
  • lead: the initial lead value, defaults to {x = 0, y = 0}
  • lerp: the initial lerp value, defaults to 0
  • max_shake_intensity: the maximum amount of camera shake, defaults to 15
  • rotation: the initial rotation of the camera, defaults to 0
  • target: the tracked object, defaults to nil; the object must have .x, .y fields. 

:update(dt[number])
  • dt: delta value passed from the main loop to update the camera 

:draw(func[function])
  • automatically wraps a function between :attach and :detach calls
  • func: the function to be drawn
1

function love.draw()
    camera:draw(drawWorld)
    drawUI()
end



:attach()
  • attaches the camera, applying all its transformations until :detach is called 

:detach()
  • detaches the camera, removing all its transformations

function love.draw()

:follow(target[table], settings[table[optional]])
  • sets a new target, follow style, lead and lerp values and resets the deadzone according to the follow style
  • target: the new target to follow, must have .x, .y fields
  • settings: a table containing target following settings, all values are optional

Possible settings:

  • follow_style: the follow style: 'lockon', 'screen', 'platformer', 'topdown' or 'topdown-tight', defaults to the already set follow style
  • lead: the new lead value, defaults to the already set lead value
  • lerp: the new lerp value, defaults to the already set lerp value 

:getPosition()
  • returns the .x, .y camera position 

:getMousePosition()
  • returns the .x, .y mouse position in world coordinates 

:getCameraCoords(x[number], y[number])
:getWorldCoords(x[number], y[number])

Because a camera has a point it looks at, a rotation and a zoom factor, it defines a coordinate system. A point now has two sets of coordinates: one defines where the point is to be found in the game world, and the other describes the position on the computer screen. The first set of coordinates is called world coordinates, the second one camera coordinates. Sometimes it is needed to convert between the two coordinate systems, for example to get the position of a mouse click in the game world in a strategy game, or to see if an object is visible on the screen.

  • returns the converted .x, .y position
  • x, y: the position in camera/world coordinates to be converted

:removeBounds()
  • removes the current set bounds 

:setBounds(left[number], top[number], right[number], down[number])
  • left, top, right, down: the boundaries the camera can’t get past, in world coordinates 

:setDeadzone(left[number], top[number], right[number], down[number])
  • defines a rectangle around the center of the screen where the target moves without camera tracking
  • left, top, right, down: the deadzone positions, in screen coordinates assuming 0, 0 is the center 

:shake(intensity[number], duration[number], settings[table[optional]])
  • intensity: the amount of pixels the camera should shake around
  • duration: the amount of seconds the camera should shake for
  • settings: a table containing camera shake settings, all values are optional

Possible settings:

  • direction: the direction in which to shake: 'vertical', 'horizontal' or 'both', defaults to 'both' 
1

camera:shake(5, 0.5, {direction = 'vertical'})



:move(x[number], y[number])
  • moves the camera by some amount, equivalent to camera.x, camera.y = camera.x + x, camera.y + y
  • x, y: the amount to move the camera by 

:moveTo(x[number], y[number])
  • x, y: the new camera position 

:rotate(angle[number])
  • rotates the camera by some angle, equivalent to camera.rotation = camera.rotation + angle
  • angle: the amount to rotate the camera by 

:rotateTo(angle[number])
  • angle: the new camera angle 

:zoom(scale[number])
  • zooms the camera by some angle, equivalent to camera.scale = camera.scale + scale
  • scale: the amount to scale the camera by 

:zoomTo(scale[number])
  • scale: the new camera scale

Attributes

.debug_draw[boolean]
  • debug drawing, shows the current follow style, lead and lerp, as well as the deadzone and current target 

.follow_style[string]
  • the camera follow_style, only set with the :follow call, otherwise the deadzone doesn’t get reset 

.lead[number]
  • how much the camera should be ahead of the moving target 

.lerp[number]
  • how much the camera should lag behind the moving target 

.max_shake_intensity[number]
  • the maximum intensity the camera can shake for 

.target[table]
  • the target the camera should follow, needs to have .x, .y fields
